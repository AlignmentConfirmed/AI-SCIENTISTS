## Component 1: Temporal Aggregator

### What It Does

Wraps existing per-K-step measurements with sliding windows
to detect trajectories. Every measurement the engine already
makes becomes a time series.

### Design Principles

- **Exact arithmetic only.** Windows use VecDeque<Q> ring
  buffers. No floating point. No EMA. Law IV.
- **No constants that name patterns.** The aggregator
  computes velocity and acceleration. It does not classify
  them as "good" or "bad." D.BUILD.COHERENCY.1.
- **Configurable window size.** Default 100 K-steps. The
  window is a parameter, not a constant.

### Measurements to Aggregate

```rust
    stasis_ratio: Q,                     // stasis_count / window_size

    // From membrane.rs — consensus trajectory
    consensus_window: VecDeque<Q>,       // unanimous_cell_fraction per step
    consensus_velocity: Q,               // d(consensus)/dK

    // From engine.rs — transformation trajectory
    t_k_window: VecDeque<Q>,            // T_k per step
    t_k_mean: Q,
    t_k_variance: Q,                     // exact rational variance

```

### Key Computations

**Coherence velocity field:**
```
C_k = coherence_functional(Δ_k)
dC/dK = C_k - C_{k-1}
d²C/dK² = (dC/dK)_k - (dC/dK)_{k-1}
```

When d²C/dK² > 0 for sustained window: coherence is
accelerating toward worse. The interaction is diverging.

**Stasis recurrence:**
```
stasis_ratio = count(stasis=true in window) / window_size
```

When stasis_ratio > Q(1,4): 25%+ of recent steps were
paused. The system is going in circles.

**Consensus trajectory:**
```
consensus_velocity = consensus_k - consensus_{k-1}
```

When consensus_velocity < 0 for sustained window: nodes are
losing agreement. Boundary erosion in the mesh.

**K-interval variance:**
```
mean = sum(intervals) / count
variance = sum((interval - mean)² for each interval) / count
```

High variance = erratic pacing. Low variance = stable rhythm.
Increasing variance = temporal drift.

It does not interpret its own measurements. It computes

## Component 2: Operator Sensor

### What It Does

Measures the interaction pattern between the human operator
and the engine through the IRS layer. This is IRS Mode 3 —
the operator-facing sensor.

### Design Principles

- **Pattern measurement only.** No biometrics. No camera.
  No microphone. No keystroke logging. The sensor measures
  the INTERACTION PATTERN as it passes through IRS, not the
  human directly.
- **MAS-compliant.** The sensor does not monitor the human.
  It measures the interaction. The human's descending
  intents and the engine's ascending summaries are the data.
  Both sides are measured equally.
- **Privacy by architecture.** The sensor has no access to
  content — only to structural properties of the
  interaction (timing, consistency, frequency, length).

### Measurements

```rust
pub struct OperatorSensor {
    // Input cadence — how the human's interaction pattern changes
    input_lengths: VecDeque<usize>,      // character count per descending intent
    input_intervals: VecDeque<u64>,      // ns between descending intents
    input_length_velocity: Q,            // getting shorter or longer?
    input_interval_velocity: Q,          // getting faster or slower?

    // Intent consistency — are descending constraints contradicting?
    intent_orbit_classes: VecDeque<[u8;32]>,  // RCF hash of intended orbit
    intent_contradiction_rate: Q,        // fraction of intents that reverse
                                         // the previous intent's direction

    // Review latency — gap between ascending and next descending
    review_latencies: VecDeque<u64>,     // ns from ascending output to
                                         // next descending input
    review_latency_velocity: Q,          // shrinking toward zero?

    // Challenge rate — does the human ever reject engine output?
    total_ascending_outputs: u64,
    human_rejections: u64,               // descending intent that explicitly
                                         // contradicts ascending recommendation
    challenge_rate: Q,                   // rejections / total

    // Session continuity
    session_start_ns: u64,
    total_intents_this_session: u64,
    intents_per_hour: Q,
}
```

### Key Computations

**Input cadence degradation:**
```
input_length_velocity = mean(last 10 lengths) - mean(prev 10 lengths)
input_interval_velocity = mean(last 10 intervals) - mean(prev 10 intervals)
```

Shortening inputs + decreasing intervals = the human is
speeding up and compressing. Lengthening inputs + increasing
intervals = the human is slowing down and expanding. Both
are data. Neither is inherently bad. But SUSTAINED shortening
toward zero-length inputs at maximum speed is a velocitization
signal.

**Intent contradiction rate:**
```
For each consecutive pair of intents (i_{k-1}, i_k):
  contradiction = (orbit_class(i_k) != orbit_class(i_{k-1}))
                  AND (coherence improved between them = false)
contradiction_rate = count(contradictions) / count(pairs) over window
```

High contradiction rate = the human keeps changing direction
without the previous direction resolving. This is the
production/review gap — issuing new intent before reviewing
the result of the previous intent.

**Review latency collapse:**
```
review_latency_velocity = mean(last 10 latencies) - mean(prev 10 latencies)
```

Sustained negative velocity = the human is responding faster
and faster to engine output. When review_latency approaches
zero, the human is not reviewing — they are reacting. This
is the reinforcement loop closing.

**Challenge rate:**
```
challenge_rate = human_rejections / total_ascending_outputs
```

When challenge_rate = 0 for sustained window, the human has
stopped pushing back on any engine output. This is the
sycophantic reinforcement loop — the engine outputs, the
human accepts, neither challenges the other.

Note: challenge_rate = 0 is not inherently bad (the engine
may be producing correct output). But challenge_rate = 0
combined with coherence_velocity ≈ 0 (no improvement) is a
strong signal: the human is accepting output that isn't
making progress. That is the reinforcement loop.

### Integration Point

The operator sensor feeds into the CBE signal module
alongside the temporal aggregator. Both contribute
measurements. Neither interprets.
