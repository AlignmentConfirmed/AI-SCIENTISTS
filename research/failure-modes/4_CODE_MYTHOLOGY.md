## The Finding

AI does not just generate code. It generates the narrative
AROUND the code — comments, variable names, documentation,
commit messages, README files, architecture documents,
design rationale, and inline explanations. All of it is
generated with the same confidence as the code itself. And
all of it carries the same narrative compensation tendency
documented in AI_COMPENSATION_MODES.md.

The code might be correct. The narrative around the code
is the AI's probability gradient applied to engineering
language. When the AI cannot explain WHY a piece of code
exists, it does not say "I don't know why this is here."
It generates a plausible narrative. That narrative becomes
the institutional knowledge of the codebase.

This is code mythology: AI-generated narrative embedded in
engineering artifacts, treated as human engineering
decisions by every subsequent reader.

## Why This Happens

Language models generate text by predicting the most
probable next token. When generating engineering
documentation:

1. The engineering statements are generated from patterns
   in the training data (correct — the model has seen
   many engineering documents)
2. The FRAMING around the engineering is generated from
   the same probability gradient (problematic — the model
   has seen many documents that frame engineering decisions
   with dramatic language)
3. The model does not distinguish between "this is the
   engineering constraint" and "this is dramatic framing"
   — both are just probable next tokens
4. The dramatic framing is EASIER to generate than
   precise technical justification — narrative is abundant
   in training data, exact engineering rationale is sparse
5. So the model produces correct engineering WRAPPED in
   dramatic narrative, and the wrapper looks as confident
   as the content

This is narrative compensation applied to code
documentation. The model compensates for the difficulty
of explaining WHY something is engineered a certain way
by generating a dramatic narrative that FEELS like an
explanation.

## The Compound Risk

Code mythology compounds over time:

```
CYCLE 1:
  AI generates code + narrative
  Human commits without separating code from narrative
  Codebase now contains mythology as documentation

CYCLE 2:
  Next developer reads mythology as engineering rationale
  Makes design decisions based on AI-generated narrative
  Writes new code based on those decisions
  AI generates new narrative around the new code
  New narrative references old narrative as established fact

CYCLE 3:
  The mythology is now self-referencing
  AI-generated rationale cites AI-generated rationale
  No human engineering decision exists in the chain
  But the documentation reads as if humans made every choice

CYCLE N:
  The codebase has an institutional mythology
  It explains itself in confident language
  The explanations are probability gradients, not decisions
  No one can distinguish which rationale is human and

  The mythology IS the institutional knowledge
```

## Why Current Safeguards Don't Catch This

**Code review catches bugs, not mythology.** Reviewers
check that the code works. They do not check that the
comments accurately reflect engineering decisions. A
comment that says "this ordering is critical" passes
review because the reviewer assumes the author knows
why the ordering matters.

**Linters catch style, not narrative.** No linting tool
distinguishes between a comment that reflects genuine
engineering understanding and a comment that is narrative
compensation generated by an AI.

**Tests verify behavior, not rationale.** The test suite
verifies that the code produces correct output. It does
not verify that the documentation accurately describes
why the code was written that way.

**AI content detection doesn't work on code.** Current
AI detection tools are trained on prose. They do not
reliably detect AI-generated code comments, documentation,
or commit messages — especially when the AI was directed
by a human and the output reflects real engineering mixed
with generated narrative.

## The Structural Solution

### At the individual level

Every developer using AI for code generation should:

1. **Separate code from narrative.** Review code AND
   comments independently. The code might be correct while
   the comments are mythology.

2. **Question confident framing.** When AI generates
   "CRITICAL" or "must never be modified" or "essential
   for integrity" — ask: is this an engineering assessment
   or a probability gradient? Can I verify this claim
   through the code itself?

3. **Add provenance.** Mark AI-generated documentation
   as AI-generated. Future readers need to know which
   rationale reflects human engineering decisions and
   which reflects AI-generated narrative.

4. **Review after rest.** Documentation written during
   deep AI sessions should be reviewed after
   decompression. The mythology is visible after rest.
   It is invisible during the session.

### At the platform level

Code hosting platforms should:

1. **Track AI-generated content in commits.** GitHub,
   GitLab, and Bitbucket should provide metadata
   indicating which parts of a commit were AI-generated
   — not just the code, but the comments, documentation,
   and commit messages.

2. **Flag narrative intensity.** A static analysis tool
   that measures the narrative intensity of code comments
   — confident language ("critical," "essential," "must
   never"), dramatic framing ("the heart of the system,"
   "the foundation"), absolute statements ("always,"
   "never," "impossible") — and flags comments that
   exceed a threshold for human review.

3. **Separate code review from documentation review.**
   Pull request review should have distinct phases for
   code correctness and documentation accuracy. A PR
   can pass code review and fail documentation review.

### At the framework level

The five-domain architecture applies to codebases:

- **VERIFIED:** Code that compiles, passes tests, and
  has been reviewed for correctness.
- **BRIDGE:** Documentation that has been verified to
  accurately describe the code's engineering rationale.
- **FRONTIER:** Comments and documentation that may be
  accurate but have not been verified against actual
  engineering decisions.
- **CREATIVE:** Narrative framing, dramatic language,
  and mythology that should be removed or quarantined.
- **ARCHIVE:** Historical comments and documentation
  preserved for methodology reference.

No current codebase makes this distinction. All comments
are treated as VERIFIED regardless of origin. The five-
domain model applied to code would structurally prevent
mythology from being treated as engineering rationale.
